## ‚ùì How it works
The project uses Sentry (or Sentry compatible alternative) for error reporting, and OpenTelemetry for logging and monitoring.

The project uses a **unified OpenTelemetry tracing architecture** that integrates both Sentry and Langfuse through a shared TracerProvider. This provides distributed tracing across services, with LLM spans exported to Langfuse (for prompt management) and all spans exported to Sentry (for application monitoring). See the [Tracing Architecture](/lonestone-boilerplate/addons/tracing-sentry#tracing-architecture) section for more details.

## Why this matters

Debugging in production can be a pain. How many times have you been stuck in a situation where you need to debug a problem in production, but:

- The client / manager has a hard time giving you the information you need (reproduction steps, context, etc.)
- You can't reproduce it locally
- You can't access or can't find the right logs (because they are written in the container, and are erased after a new deployment)
- this list goes on...

Good reporting and monitoring tools can save you hours, days or even weeks of debugging or performance tuning. Thanks to the rise of OTEL, they got easier to setup and use.

## üí° Why Sentry and OpenTelemetry

Sentry is the de facto standard for error reporting. It's UX is great, and it has numerous integrations with other tools and frameworks (even niche ones like Unity3D).

The possibility of self-hosting Sentry is another important factor.

OpenTelemetry has become the standard for logging and monitoring, and Sentry is compatible with it for tracing. There is always the possibility to use another tool for logging and tracing, like Signoz, due their compatibility with OTEL.
